1. ENUMS (Tipos enumerados)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Role             → CLIENT | PROVIDER | ADMIN
                 Define el tipo de usuario en el sistema.

ProviderStatus   → PENDING | APPROVED | REJECTED
                 Estado de verificación de un provider en una especialidad.
                 El admin aprueba o rechaza según los documentos subidos.

ServiceStatus    → SEARCHING | ASSIGNED | ON_WAY | IN_PROGRESS | COMPLETED | CANCELLED
                 Ciclo de vida de un servicio desde que el cliente lo solicita
                 hasta que se completa o cancela.

MessageType      → TEXT | IMAGE | AUDIO
                 Tipo de mensaje dentro del chat de un servicio.

CallType         → VOICE | VIDEO
                 Tipo de llamada entre cliente y provider.

CallStatus       → INITIATED | ACCEPTED | REJECTED | ENDED | MISSED
                 Estado de una llamada en tiempo real.

RechargeStatus   → PENDING | APPROVED | REJECTED
                 Estado de una recarga de saldo. Se usa en dos modelos:
                   - Recharge         → recarga del ClientWallet del cliente
                   - ProviderPurchase → recarga del ProviderWallet del provider

WithdrawalStatus → PENDING | APPROVED | REJECTED
                 Estado de una solicitud de retiro de ganancias del provider.

MediaType        → IMAGE | VIDEO
                 Tipo de media usado en History, PostMedia, Reel y Anuncio.

2. MODELOS Y SUS CAMPOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
COUNTRY
────────────────────────────────────────
Propósito: Catálogo de países gestionado por el admin.
El frontend usa esta tabla para poblar el combo box de selección
de país al registrarse. Define el código de marcado y la moneda
local de cada país. Se llena inicialmente con el seed.

Campos:
  - iso       → Código ISO del país, clave primaria (ej: BO, US, PE)
  - dialCode  → Código de marcado telefónico (ej: +591, +1, +51)
  - name      → Nombre del país (ej: Bolivia, Estados Unidos)
  - currency  → Moneda local del país (ej: BOB, USD, PEN)
  - active    → false = no aparece en el combo box del frontend
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - clientProfiles   → Perfiles de clientes de este país
  - providerProfiles → Perfiles de providers de este país
  - providerServices → Especialidades de providers registradas en este país


────────────────────────────────────────
SYSTEMCONFIG
────────────────────────────────────────
Propósito: Configuración global del sistema. Solo existe UN registro.
El admin lo actualiza cuando necesita ajustar parámetros globales.

Campos:
  - commissionRate    → Comisión que cobra la plataforma por servicio (0.15 = 15%)
  - maxConcurrentJobs → Máximo de trabajos simultáneos por especialidad (ej: 2)
  - updatedAt         → Última actualización

Relaciones: Ninguna (tabla de configuración independiente)


────────────────────────────────────────
EXCHANGERATE
────────────────────────────────────────
Propósito: Tipos de cambio para convertir paquetes de USD a la moneda
local del cliente o provider. El admin actualiza las tasas desde el panel.
Siempre fromCurrency = "USD".

Campos:
  - fromCurrency → Moneda origen (siempre "USD")
  - toCurrency   → Moneda destino (ej: "BOB", "PEN", "ARS")
  - rate         → Tasa de conversión (ej: 6.96 para USD → BOB)
  - createdAt    → Fecha de creación
  - updatedAt    → Última actualización

Restricción: @@unique([fromCurrency, toCurrency])
             No puede haber dos tasas para el mismo par de monedas.

Relaciones: Ninguna (tabla de referencia consultada en código)

Ejemplo de datos:
  USD → BOB  rate: 6.96
  USD → PEN  rate: 3.70
  USD → ARS  rate: 900.00

Uso en el flujo de recargas:
  Package.amountUsd × ExchangeRate.rate = monto en moneda local
  Ej: 10 USD × 6.96 = 69.60 BOB acreditados al ClientWallet


────────────────────────────────────────
SERVICETYPE
────────────────────────────────────────
Propósito: Catálogo de tipos de servicio que ofrece la plataforma.
El admin puede crear, activar o desactivar categorías sin tocar el código.

Campos:
  - name        → Nombre único (ej: "Gasfitería", "Electricidad")
  - icon        → URL o nombre del ícono para la app
  - description → Descripción del tipo de servicio
  - active      → false = no aparece en la app para nuevos servicios
  - createdAt   → Fecha de creación
  - updatedAt   → Última actualización

Relaciones:
  - providerServices → Providers registrados en esta categoría
  - services         → Servicios solicitados de esta categoría


────────────────────────────────────────
USER
────────────────────────────────────────
Propósito: Modelo base SOLO para autenticación. Cualquier usuario
(cliente, provider o admin) tiene un User. Los datos personales
viven en ClientProfile o ProviderProfile.

Campos:
  - email               → Correo para iniciar sesión (único, opcional para Google)
  - username            → Elegido por el usuario al registrarse.
                          Solo letras y números. Si ya existe, el backend
                          sugiere 3 alternativas disponibles.
  - password            → Hash bcrypt de la contraseña (vacío para cuentas Google)
  - role                → CLIENT | PROVIDER | ADMIN
  - isActive            → false = cuenta bloqueada por el admin
  - fcmToken            → Token Firebase para notificaciones push
  - resetPasswordToken  → Hash del código de recuperación de contraseña
  - resetPasswordExpiry → Expiración del código de recuperación (15 minutos)
  - createdAt           → Fecha de creación
  - updatedAt           → Última actualización

Login soporta 3 opciones:
  1. email + contraseña
  2. username + contraseña
  3. Google OAuth (flujo de 2 pasos para usuarios nuevos)

Relaciones:
  - clientProfile    → 1 a 1 opcional. Existe si role = CLIENT
  - providerProfile  → 1 a 1 opcional. Existe si role = PROVIDER
  - providerServices → 1 a muchos. Especialidades si role = PROVIDER
  - sentMessages     → Mensajes enviados en servicios
  - clientServices   → Servicios solicitados (solo clientes)
  - ratings          → Calificaciones dadas (solo clientes)
  - historyViews     → Historias vistas (cualquier rol)
  - savedProviders   → Especialidades guardadas como favoritas
  - postLikes        → Likes dados a publicaciones
  - postComments     → Comentarios en publicaciones
  - reelLikes        → Likes dados a reels
  - reelComments     → Comentarios en reels
  - savedBy          → Publicaciones guardadas como favoritas


────────────────────────────────────────
CLIENTPROFILE
────────────────────────────────────────
Propósito: Datos personales del cliente. Se crea automáticamente
al registrarse como CLIENT. Tiene su propio ClientWallet.

Campos:
  - userId         → FK a User (1 a 1)
  - countryIso     → FK a Country — define la moneda del ClientWallet.
                     Se elige al registrarse y NO cambia nunca.
  - firstName      → Nombre (nullable — se completa en el paso 2 del registro)
  - lastName       → Apellido (nullable — se completa en el paso 2 del registro)
  - avatar         → URL pública de la foto de perfil en Cloudinary
  - avatarPublicId → public_id de Cloudinary para reemplazar o eliminar la imagen
  - referralCode   → Código único para referir a otros usuarios
  - lastLoginAt    → Último inicio de sesión
  - createdAt      → Fecha de creación
  - updatedAt      → Última actualización

Relaciones:
  - clientWallet → Billetera para pagar servicios (1 a 1)
  - contacts     → Números de contacto del cliente (1 a N)
  - user         → Pertenece a un User (1 a 1)
  - country      → Pertenece a un Country (define la moneda)


────────────────────────────────────────
PROVIDERPROFILE
────────────────────────────────────────
Propósito: Datos personales del provider. UN solo perfil sin importar
cuántas especialidades tenga. Se crea al registrarse como PROVIDER.
Las especialidades viven en ProviderService.

Campos:
  - userId         → FK a User (1 a 1)
  - countryIso     → FK a Country — identifica de qué país es el provider.
                     Solo informativo. No define moneda (eso lo hace ProviderService).
  - firstName      → Nombre (nullable — se completa en el paso 2 del registro)
  - lastName       → Apellido (nullable — se completa en el paso 2 del registro)
  - avatar         → URL pública de la foto de perfil en Cloudinary
  - avatarPublicId → public_id de Cloudinary para reemplazar o eliminar la imagen
  - referralCode   → Código único para referir a otros usuarios
  - lastLoginAt    → Último inicio de sesión
  - createdAt      → Fecha de creación
  - updatedAt      → Última actualización

Relaciones:
  - user    → Pertenece a un User (1 a 1)
  - country → Pertenece a un Country (informativo)


────────────────────────────────────────
CONTACT
────────────────────────────────────────
Propósito: Números de contacto del cliente. Un cliente puede tener
múltiples números (máx 3). Separado de ClientProfile para permitir
varios registros por cliente.

Campos:
  - clientProfileId → FK a ClientProfile
  - phone           → Número de teléfono único en la plataforma
  - createdAt       → Fecha de creación
  - updatedAt       → Última actualización

Relaciones:
  - clientProfile → Pertenece a un ClientProfile


────────────────────────────────────────
PROVIDERCONTACT
────────────────────────────────────────
Propósito: Números de contacto del provider por especialidad.
Una especialidad puede tener múltiples números. Separado de
ProviderService para permitir varios registros por especialidad.

Campos:
  - providerServiceId → FK a ProviderService
  - phone             → Número de teléfono único en la plataforma
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService


────────────────────────────────────────
PROVIDERSERVICE
────────────────────────────────────────
Propósito: Datos OPERATIVOS de una especialidad del provider. Un usuario
con role PROVIDER puede tener MÚLTIPLES registros ProviderService, uno
por cada tipo de servicio en el que está habilitado.

Ejemplo: Un provider puede tener Gasfitería y Electricidad, cada uno
con su propio estado de verificación, calificación, wallet y disponibilidad.
Igual que Uber — un solo login, múltiples modalidades de trabajo.

Restricción: @@unique([userId, serviceTypeId]) → No puede postularse
             dos veces a la misma especialidad.

Campos:
  - userId        → Provider al que pertenece esta especialidad
  - serviceTypeId → Tipo de servicio (Gasfitería, Electricidad, etc.)
  - countryIso    → FK a Country — define la moneda del ProviderWallet.
                    Se elige al postularse y NO cambia nunca.
  - status        → PENDING | APPROVED | REJECTED (el admin verifica documentos)
  - available     → true = disponible para recibir nuevos trabajos ahora mismo
  - rating        → Promedio de calificaciones (0.0 a 5.0)
  - totalRatings  → Cantidad total de calificaciones recibidas
  - latitude      → Ubicación actual para encontrar el provider más cercano
  - longitude     → Ubicación actual
  - createdAt     → Fecha de creación
  - updatedAt     → Última actualización

Relaciones:
  - user            → Pertenece a un User
  - serviceType     → Pertenece a un ServiceType
  - country         → Pertenece a un Country (define moneda del wallet)
  - providerWallet  → Billetera de ganancias (1 a 1)
  - profile         → Perfil de presentación (1 a 1)
  - documents       → Documentos de verificación (1 a N)
  - providerContacts→ Números de contacto de esta especialidad (1 a N)
  - services        → Servicios atendidos (1 a N)
  - histories       → Historias publicadas (1 a N)
  - posts           → Publicaciones tipo Facebook (1 a N)
  - reels           → Reels cortos tipo TikTok (1 a N)
  - anuncios        → Anuncios pagados (1 a N)
  - savedBy         → Usuarios que guardaron esta especialidad como favorita (1 a N)


────────────────────────────────────────
PROVIDERSERVICEPROFILE
────────────────────────────────────────
Propósito: Datos de PRESENTACIÓN de una especialidad. Contiene todo
lo que el cliente ve y lo que el provider puede editar libremente.
Se crea al postularse junto con ProviderService.

Separación de responsabilidades:
  ProviderService        → datos operativos (estado, ubicación, rating, moneda)
  ProviderServiceProfile → datos de presentación (bio, foto, nombre personalizado)

Campos:
  - providerServiceId    → FK a ProviderService (1 a 1)
  - bio                  → Descripción profesional para esta especialidad
  - experience           → Años de experiencia
  - coverageRadius       → Radio máximo de cobertura en kilómetros
  - commercialName       → Nombre del negocio o marca (opcional)
  - customServiceName    → Nombre personalizado del servicio
                           Ej: "Plomería Express", "Electricista 24h"
  - servicePhoto         → URL pública de la foto del servicio en Cloudinary
  - servicePhotoPublicId → public_id de Cloudinary para reemplazar o eliminar
  - createdAt            → Fecha de creación
  - updatedAt            → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (1 a 1)

PROVIDERDOCUMENT

  - publicId          → public_id de Cloudinary para eliminar o reemplazar
  - name              → Nombre descriptivo (ej: "Certificado de electricista")
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService


────────────────────────────────────────
CLIENTWALLET
────────────────────────────────────────
Propósito: Billetera del cliente. Solo para pagar servicios.
Recibe recargas aprobadas por el admin.

Campos:
  - clientProfileId → Cliente dueño de esta billetera (FK único)
  - currency        → Moneda fija (ej: BOB, USD, PEN).
                      Se asigna al registrarse según countryIso y NUNCA cambia.
  - balance         → Saldo disponible para pagar servicios
  - createdAt       → Fecha de creación
  - updatedAt       → Última actualización

Relaciones:
  - clientProfile → Pertenece a un ClientProfile (1 a 1)
  - recharges     → Historial de recargas


────────────────────────────────────────
PROVIDERWALLET
────────────────────────────────────────
Propósito: Billetera del provider por especialidad. Acumula ganancias
de servicios completados. Cada especialidad tiene su propia wallet
independiente. También financia los anuncios del provider.

Campos:
  - providerServiceId → Especialidad dueña de esta billetera (FK único)
  - currency          → Moneda fija (ej: BOB, USD, PEN).
                        Se asigna al postularse según countryIso y NUNCA cambia.
  - balance           → Ganancias acumuladas pendientes de retiro
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (1 a 1)
  - withdrawals     → Historial de retiros
  - purchases       → Historial de recargas (ProviderPurchase)
  - anuncios        → Anuncios financiados desde esta billetera


────────────────────────────────────────
PACKAGE
────────────────────────────────────────
Propósito: Paquetes de recarga que el admin crea. Tanto clientes como
providers pueden comprar paquetes para recargar su wallet.
El precio base está en USD y se convierte automáticamente a la moneda
local del usuario usando la tabla ExchangeRate.

Campos:
  - name      → Nombre del paquete (ej: "Paquete Básico", "Paquete Pro")
  - amountUsd → Precio base en USD (ej: 10.00). Se convierte al mostrar.
  - active    → false = no aparece para los usuarios
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - recharges         → Recargas de clientes que usaron este paquete
  - providerPurchases → Compras de providers que usaron este paquete


────────────────────────────────────────
RECHARGE
────────────────────────────────────────
Propósito: Registro de cada recarga de saldo que hace un cliente.
Flujo: Cliente elige Package → Recharge (PENDING) →
admin aprueba → ClientWallet.balance += package.amountUsd × ExchangeRate.rate

Campos:
  - clientWalletId → Wallet del cliente que recarga
  - packageId      → Paquete elegido por el cliente
  - status         → PENDING | APPROVED | REJECTED
  - createdAt      → Fecha de creación
  - updatedAt      → Última actualización

Relaciones:
  - clientWallet → Pertenece a un ClientWallet
  - package      → Pertenece a un Package


────────────────────────────────────────
PROVIDERPURCHASE
────────────────────────────────────────
Propósito: Registro de cada recarga de saldo que hace un provider.
El provider compra un paquete para acreditar saldo en el ProviderWallet
de una especialidad específica.
Flujo: Provider elige Package → ProviderPurchase (PENDING) →
admin aprueba → ProviderWallet.balance += package.amountUsd × ExchangeRate.rate

Campos:
  - providerWalletId → Wallet del provider que recarga
  - packageId        → Paquete elegido por el provider
  - status           → PENDING | APPROVED | REJECTED (usa RechargeStatus)
  - createdAt        → Fecha de creación
  - updatedAt        → Última actualización

Nota: Usa el mismo enum RechargeStatus que Recharge porque los estados
      son idénticos. Tabla separada porque el FK apunta a ProviderWallet
      en lugar de ClientWallet.

Relaciones:
  - providerWallet → Pertenece a un ProviderWallet
  - package        → Pertenece a un Package


────────────────────────────────────────
WITHDRAWAL
────────────────────────────────────────
Propósito: Solicitud de retiro de ganancias del provider.
El provider solicita retirar el saldo de una especialidad específica.
Flujo: Provider solicita → Withdrawal (PENDING) →
admin aprueba → ProviderWallet.balance -= amount

Campos:
  - providerWalletId → Wallet de la especialidad de donde se retira
  - amount           → Monto a retirar
  - bankName         → Nombre del banco destino
  - accountNumber    → Número de cuenta bancaria destino
  - status           → PENDING | APPROVED | REJECTED
  - createdAt        → Fecha de creación
  - updatedAt        → Última actualización

Relaciones:
  - providerWallet → Pertenece a un ProviderWallet


────────────────────────────────────────
HISTORY
────────────────────────────────────────
Propósito: Historia que el provider sube para mostrar su trabajo
a los clientes, estilo WhatsApp/Instagram Stories.
Expira automáticamente a las 24 horas — un cron job la elimina
y borra el media de Cloudinary usando mediaPublicId.

Campos:
  - providerServiceId → Especialidad que publica la historia
  - mediaUrl          → URL pública del media en Cloudinary
  - mediaPublicId     → public_id de Cloudinary para eliminar al expirar
  - mediaType         → IMAGE | VIDEO
  - caption           → Texto opcional sobre la historia
  - expiresAt         → createdAt + 24h — referencia para el cron job
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService
  - historyViews    → Registro de quién la vio (1 a N)


────────────────────────────────────────
HISTORYVIEW
────────────────────────────────────────
Propósito: Registra cada vez que un usuario ve una historia.
Sirve para mostrar el conteo de vistas y evitar duplicados
del mismo usuario.

Campos:
  - historyId → Historia que se vio
  - userId    → Usuario que la vio (cualquier rol)
  - viewedAt  → Momento en que se vio

Restricción: @@unique([historyId, userId])
             Un usuario no puede contar como vista dos veces.

Relaciones:
  - history → Pertenece a una History
  - user    → Pertenece a un User


────────────────────────────────────────
POST
────────────────────────────────────────
Propósito: Publicación permanente del provider, estilo Facebook.
Puede tener texto y múltiples fotos o videos en carrusel.
Los usuarios pueden dar like y comentar.

Campos:
  - providerServiceId → Especialidad que publica
  - content           → Texto de la publicación (opcional)
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService
  - media           → Fotos y videos adjuntos en carrusel (1 a N)
  - likes           → Likes de usuarios (1 a N)
  - comments        → Comentarios de usuarios (1 a N)
  - savedPosts      → Usuarios que guardaron esta publicación (1 a N)


────────────────────────────────────────
POSTMEDIA
────────────────────────────────────────
Propósito: Media (foto o video) adjunta a una publicación.
Un Post puede tener múltiples medias ordenadas como carrusel.

Campos:
  - postId        → Publicación a la que pertenece
  - mediaUrl      → URL Cloudinary
  - mediaPublicId → public_id para eliminar de Cloudinary
  - mediaType     → IMAGE | VIDEO
  - order         → Orden en el carrusel (1, 2, 3...)
  - createdAt     → Fecha de creación
  - updatedAt     → Última actualización

Relaciones:
  - post → Pertenece a un Post


────────────────────────────────────────
POSTLIKE

Campos:
  - postId    → Publicación que recibió el like
  - userId    → Usuario que dio el like
  - createdAt → Fecha del like

Relaciones:
  - post → Pertenece a un Post

────────────────────────────────────────
POSTCOMMENT
────────────────────────────────────────

Campos:
  - postId    → Publicación comentada
  - userId    → Usuario que comentó
  - content   → Texto del comentario
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - post → Pertenece a un Post
  - user → Pertenece a un User


────────────────────────────────────────
SAVEDPOST
────────────────────────────────────────
Propósito: Tabla intermedia para que un usuario guarde publicaciones

Un usuario no puede guardar la misma publicación dos veces.

Campos:
  - userId    → Usuario que guardó
  - postId    → Publicación guardada
  - createdAt → Fecha en que se guardó

Restricción: @@unique([userId, postId])

Relaciones:
  - user → Pertenece a un User
  - post → Pertenece a un Post


────────────────────────────────────────
SAVEDPROVIDER
────────────────────────────────────────
Propósito: Tabla intermedia para que un usuario guarde especialidades
de providers como favoritas, estilo "contacto favorito" de Uber.
Un usuario no puede guardar la misma especialidad dos veces.

Nota: Apunta a User (no ClientProfile) para que cualquier rol
      pueda guardar providers a futuro.

Campos:
  - clientId          → Usuario que guardó
  - providerServiceId → Especialidad guardada
  - createdAt         → Fecha en que se guardó
  - updatedAt         → Última actualización

Restricción: @@unique([clientId, providerServiceId])

Relaciones:
  - client          → Pertenece a un User
  - providerService → Pertenece a un ProviderService


────────────────────────────────────────
REEL
────────────────────────────────────────
Propósito: Video corto vertical del provider para mostrar su trabajo,
estilo TikTok / Instagram Reels. Independiente de Post —
tiene su propia sección en la app con thumbnail y contador de vistas.

Campos:
  - providerServiceId → Especialidad que sube el reel
  - videoUrl          → URL Cloudinary del video
  - videoPublicId     → public_id para eliminar de Cloudinary
  - thumbnailUrl      → URL Cloudinary del thumbnail (primer frame)
  - thumbnailPublicId → public_id del thumbnail para eliminar
  - caption           → Texto opcional sobre el reel
  - duration          → Duración en segundos (máx recomendado: 60)
  - views             → Contador de reproducciones
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Diferencia con Post:
  Post  → publicación permanente, texto + carrusel de fotos/videos
  Reel  → video corto vertical con thumbnail, sección propia, contador de vistas

Relaciones:
  - providerService → Pertenece a un ProviderService
  - likes           → Likes de usuarios (1 a N)
  - comments        → Comentarios de usuarios (1 a N)

REELLIKE
────────────────────────────────────────
Propósito: Like de un usuario a un reel.
Un usuario solo puede dar un like por reel.

Campos:
  - reelId    → Reel que recibió el like
  - userId    → Usuario que dio el like
  - createdAt → Fecha del like

Restricción: @@unique([reelId, userId])

Relaciones:
  - reel → Pertenece a un Reel
  - user → Pertenece a un User


────────────────────────────────────────
REELCOMMENT
────────────────────────────────────────
Propósito: Comentario de un usuario en un reel.

Campos:
  - reelId    → Reel comentado
  - userId    → Usuario que comentó
  - content   → Texto del comentario
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - reel → Pertenece a un Reel
  - user → Pertenece a un User


────────────────────────────────────────
ANUNCIO
────────────────────────────────────────
Propósito: Anuncio pagado por el provider para destacar su especialidad
en los resultados de búsqueda. Funciona como prepago — el saldo se
descuenta del ProviderWallet al crear el anuncio.

Flujo:
  Provider crea anuncio con budget → ProviderWallet.balance -= budget
  → Anuncio activo hasta endDate
  → Cron job desactiva el anuncio al llegar endDate (active = false)
  → Providers con anuncio activo aparecen primero en búsquedas

Campos:
  - providerWalletId  → Wallet de donde se descontó el saldo
  - providerServiceId → Especialidad que se promociona
  - title             → Título del anuncio
  - description       → Texto descriptivo (opcional)
  - mediaUrl          → URL Cloudinary del media del anuncio (opcional)
  - mediaPublicId     → public_id para eliminar de Cloudinary
  - mediaType         → IMAGE | VIDEO (opcional si no hay media)
  - budget            → Saldo descontado al crear
  - startDate         → Fecha de inicio
  - endDate           → Fecha de fin (cron job desactiva al llegar)
  - active            → false = expirado o desactivado manualmente
  - impressions       → Veces que se mostró en búsquedas
  - clicks            → Veces que tocaron el anuncio
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Nota: providerWalletId y providerServiceId son del mismo provider
      pero cumplen roles distintos:
        providerWalletId  → rastro financiero (de dónde salió el dinero)
        providerServiceId → rastro comercial (qué se promociona)

Relaciones:
  - providerWallet  → Pertenece a un ProviderWallet (origen del pago)
  - providerService → Pertenece a un ProviderService (lo que se promociona)


────────────────────────────────────────
SERVICE
────────────────────────────────────────
Propósito: Núcleo del negocio. Representa un servicio solicitado
por un cliente. Conecta al cliente con un ProviderService específico.

Flujo de estados:
  SEARCHING   → Cliente solicita, sistema busca provider disponible
  ASSIGNED    → Provider aceptó el trabajo
  ON_WAY      → Provider va en camino al domicilio
  IN_PROGRESS → Provider está realizando el trabajo
  COMPLETED   → Trabajo finalizado
  CANCELLED   → Servicio cancelado

Campos:
  - clientId          → Cliente que solicitó
  - providerServiceId → Especialidad asignada (null hasta que se asigne)
  - serviceTypeId     → Tipo de servicio solicitado
  - description       → Descripción del problema o trabajo
  - address           → Dirección donde se realiza
  - latitude          → Coordenadas para el mapa
  - longitude         → Coordenadas para el mapa
  - status            → Estado actual del servicio
  - price             → Precio acordado (null en estado SEARCHING)
  - commission        → Comisión cobrada (price × SystemConfig.commissionRate)
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - client          → Pertenece a un User (cliente)
  - providerService → Pertenece a un ProviderService (opcional hasta asignación)
  - serviceType     → Pertenece a un ServiceType
  - messages        → Chat del servicio (1 a N)
  - calls           → Llamadas realizadas (1 a N)
  - rating          → Calificación del cliente al finalizar (1 a 1)


────────────────────────────────────────
MESSAGE
────────────────────────────────────────
Propósito: Mensajes del chat dentro de un servicio activo.
Cliente y provider se comunican durante el servicio.

Campos:
  - serviceId → Servicio al que pertenece
  - senderId  → Usuario que envió el mensaje
  - content   → Texto o URL del archivo (imagen/audio)
  - type      → TEXT | IMAGE | AUDIO
  - read      → true = el receptor ya lo leyó
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - service → Pertenece a un Service
  - sender  → Pertenece a un User


────────────────────────────────────────
CALL
────────────────────────────────────────
Propósito: Registro de llamadas de voz o video entre cliente
y provider durante un servicio.

Campos:
  - serviceId  → Servicio en el que ocurrió la llamada
  - callerId   → Usuario que inició la llamada
  - receiverId → Usuario que recibe la llamada
  - type       → VOICE | VIDEO
  - status     → INITIATED | ACCEPTED | REJECTED | ENDED | MISSED
  - startedAt  → Momento en que el receptor aceptó
  - endedAt    → Momento en que terminó
  - duration   → Duración en segundos (calculado al terminar)
  - createdAt  → Fecha de creación
  - updatedAt  → Última actualización

Relaciones:
  - service → Pertenece a un Service


────────────────────────────────────────
RATING
────────────────────────────────────────
Propósito: Calificación que el cliente da al provider después
de completar un servicio. Solo una calificación por servicio.
Al crearse recalcula ProviderService.rating automáticamente.

Fórmula: rating = (rating × totalRatings + score) / (totalRatings + 1)

Campos:
  - serviceId → Servicio calificado (único — un solo Rating por servicio)
  - clientId  → Cliente que calificó
  - score     → Puntuación del 1 al 5
  - comment   → Comentario opcional
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - service → Pertenece a un Service (1 a 1)
  - client  → Pertenece a un User (cliente)

4. FLUJOS PRINCIPALES DEL NEGOCIO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

REGISTRO CLIENTE (2 pasos):
  Paso 1 → POST /auth/register
    User (CLIENT) + username + countryIso
    → ClientProfile (countryIso, firstName: null, lastName: null)
    → ClientWallet (currency = Country.currency, balance: 0)
  Paso 2 → PUT /profile/complete
    → ClientProfile.firstName, lastName
    → Contact[] (teléfonos opcionales, máx 3)

REGISTRO PROVIDER (2 pasos):
  Paso 1 → POST /auth/register/technician
    User (PROVIDER) + username + countryIso
    → ProviderProfile (countryIso, firstName: null, lastName: null)
    (sin wallet — se crea al postularse a especialidad)
  Paso 2 → PUT /profile/complete
    → ProviderProfile.firstName, lastName

REGISTRO CON GOOGLE (flujo unificado):
  POST /auth/google → si email existe: login directo
                   → si email nuevo: devuelve tempToken + datos de Google
  POST /auth/google/complete → username + countryIso + role
    → igual que registro normal pero firstName/lastName vienen de Google
    → no requiere Paso 2

LOGIN (3 opciones):
  1. email + contraseña    → POST /auth/login
  2. username + contraseña → POST /auth/login
  3. Google OAuth          → POST /auth/google
  → Actualiza lastLoginAt en ClientProfile o ProviderProfile

POSTULACIÓN A ESPECIALIDAD:
  Provider elige ServiceType + countryIso
  → ProviderService (PENDING)
  → ProviderWallet (currency = Country.currency, balance: 0)
  → ProviderServiceProfile (bio, experience, etc.)
  → ProviderDocument[] (certificados)
  → ProviderContact[] (teléfonos de esta especialidad)
  Admin aprueba → ProviderService.status = APPROVED

RECARGA DE SALDO (CLIENTE):
  GET /packages → cliente elige paquete
  POST /wallet/recharge → Recharge (PENDING)
  Admin aprueba:
    tasa = ExchangeRate WHERE fromCurrency='USD' AND toCurrency=wallet.currency
    ClientWallet.balance += package.amountUsd × tasa
  Ej: 10 USD × 6.96 = 69.60 BOB

RECARGA DE SALDO (PROVIDER):
  GET /packages → provider elige paquete + especialidad
  POST /wallet/purchase → ProviderPurchase (PENDING)
  Admin aprueba:
    tasa = ExchangeRate WHERE fromCurrency='USD' AND toCurrency=wallet.currency
    ProviderWallet.balance += package.amountUsd × tasa

SOLICITUD DE SERVICIO:
  Cliente crea Service (SEARCHING)
  Sistema busca ProviderService donde:
    status = APPROVED AND available = true
    serviceTypeId = el solicitado
    trabajos activos < SystemConfig.maxConcurrentJobs
    distancia < coverageRadius (por lat/lng)
  Provider acepta → ASSIGNED → ON_WAY → IN_PROGRESS → COMPLETED

PAGO AL COMPLETAR:
  ClientWallet.balance -= price
  ProviderWallet.balance += (price - commission)
  commission = price × SystemConfig.commissionRate

CALIFICACIÓN:
  Service COMPLETED → Cliente crea Rating (score 1-5)
  ProviderService.rating = (rating × totalRatings + score) / (totalRatings + 1)
  ProviderService.totalRatings += 1

RETIRO DE GANANCIAS:
  Provider solicita Withdrawal (amount, bankName, accountNumber)
  Admin aprueba → ProviderWallet.balance -= amount

CONTENIDO DEL PROVIDER:
  Historia → expira en 24h → cron job la elimina + borra media de Cloudinary
  Post     → permanente, con carrusel de fotos/videos, likes y comentarios
  Reel     → video corto vertical, thumbnail, contador de vistas, likes y comentarios

ANUNCIO PAGADO:
  Provider crea Anuncio con budget → ProviderWallet.balance -= budget
  Anuncio activo hasta endDate → aparece destacado en búsquedas
  Cron job desactiva al llegar endDate (active = false)
  Métricas: impressions (veces mostrado) + clicks (veces tocado)

RECUPERAR CONTRASEÑA:
  Usuario ingresa email → código de 6 dígitos al correo (válido 15 min)
  Máximo 5 intentos fallidos → código invalidado
  Usuario ingresa código → nueva contraseña

5. DECISIONES DE DISEÑO IMPORTANTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. MONEDA FIJA EN WALLETS
   ClientWallet.currency y ProviderWallet.currency se asignan
   al momento de crear el wallet y NUNCA cambian.
   Esto evita inconsistencias si el admin modifica Country.currency
   en el futuro. El wallet siempre refleja la moneda original.

2. PAQUETES EN USD CON CONVERSIÓN AUTOMÁTICA
   Package.amountUsd siempre en USD.
   ExchangeRate convierte al momento de acreditar.
   Un solo catálogo de paquetes sirve para todos los países.

3. REGISTRO EN DOS PASOS
   Paso 1 crea credenciales + país + wallet (mínimo para operar).
   Paso 2 completa nombre, apellido y teléfonos.
   Esto permite que el usuario empiece a explorar la app
   antes de completar su perfil.

4. CONTENIDO LIGADO A ESPECIALIDAD, NO AL PERFIL
   History, Post, Reel y Anuncio apuntan a ProviderService,
   no a ProviderProfile. Esto permite que cada especialidad
   tenga su propio portafolio de contenido independiente.
   Ej: Juan el Electricista y Juan el Gasfitero tienen
   historias, posts y reels separados.

5. SAVEDPROVIDER Y SAVEDPOST USAN USER (NO CLIENTPROFILE)
   Apuntan a User para que cualquier rol pueda guardar
   favoritos en el futuro, no solo clientes.

6. PRICE Y COMMISSION SON NULLABLE EN SERVICE
   Ambos son Float? porque en SEARCHING el precio aún
   no se ha acordado entre cliente y provider.

7. WALLETS SEPARADOS POR ROL
   ClientWallet  → pertenece a ClientProfile
   ProviderWallet → pertenece a ProviderService (uno por especialidad)
   No hay tabla Wallet genérica — evita nulls innecesarios.

8. ANUNCIO TIENE DOS FKs AL MISMO PROVIDER
   providerWalletId  → rastro financiero (de dónde salió el dinero)
   providerServiceId → rastro comercial (qué especialidad se promociona)
   Ambas son necesarias por semántica y auditoría.
