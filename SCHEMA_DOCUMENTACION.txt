1. ENUMS (Tipos enumerados)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ProviderStatus   → PENDING | APPROVED | REJECTED
                 Estado de verificación de un usuario en una especialidad.
                 El admin aprueba o rechaza según los documentos subidos.

ServiceStatus    → SEARCHING | ASSIGNED | ON_WAY | IN_PROGRESS | COMPLETED | CANCELLED
                 Ciclo de vida de un servicio desde que el usuario lo solicita
                 hasta que se completa o cancela.

MessageType      → TEXT | IMAGE | AUDIO
                 Tipo de mensaje dentro del chat de un servicio.

CallType         → VOICE | VIDEO
                 Tipo de llamada entre usuario y provider.

CallStatus       → INITIATED | ACCEPTED | REJECTED | ENDED | MISSED
                 Estado de una llamada en tiempo real.

RechargeStatus   → PENDING | APPROVED | REJECTED
                 Estado de una recarga de saldo. Se usa en dos modelos:
                   - Recharge         → recarga del Wallet del usuario
                   - ProviderPurchase → recarga del ProviderWallet del provider

WithdrawalStatus → PENDING | APPROVED | REJECTED
                 Estado de una solicitud de retiro de ganancias del provider.

MediaType        → IMAGE | VIDEO
                 Tipo de media usado en History, PostMedia, Reel y Anuncio.

────────────────────────────────────────
COUNTRY
────────────────────────────────────────
Propósito: Catálogo de países gestionado por el admin.
El frontend usa esta tabla para poblar el combo box de selección
de país al registrarse. Define el código de marcado telefónico
(dialCode) y la moneda local de cada país.
El dialCode lo usa el frontend para armar el teléfono de login:
  Country.dialCode + número = User.phone  (ej: +591 + 70012345)

Campos:
  - iso       → Código ISO del país, clave primaria (ej: BO, US, PE)
  - dialCode  → Código de marcado telefónico (ej: +591, +1, +51)
  - name      → Nombre del país (ej: Bolivia, Estados Unidos)
  - currency  → Moneda local del país (ej: BOB, USD, PEN)
  - active    → false = no aparece en el combo box del frontend
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - profiles         → Usuarios registrados en este país (1 a N)
  - providerServices → Especialidades registradas en este país (1 a N)


────────────────────────────────────────
SYSTEMCONFIG
────────────────────────────────────────
Propósito: Configuración global del sistema. Solo existe UN registro.
El admin lo actualiza cuando necesita ajustar parámetros globales.

Campos:
  - commissionRate    → Comisión que cobra la plataforma por servicio (0.15 = 15%)
  - maxConcurrentJobs → Máximo de trabajos simultáneos por especialidad (ej: 2)
  - updatedAt         → Última actualización

Relaciones: Ninguna (tabla de configuración independiente)


────────────────────────────────────────
EXCHANGERATE
────────────────────────────────────────
Propósito: Tipos de cambio para convertir paquetes de USD a la moneda
local del usuario. El admin actualiza las tasas desde el panel.
Siempre fromCurrency = "USD".

Campos:
  - fromCurrency → Moneda origen (siempre "USD")
  - toCurrency   → Moneda destino (ej: "BOB", "PEN", "ARS")
  - rate         → Tasa de conversión (ej: 6.96 para USD → BOB)
  - createdAt    → Fecha de creación
  - updatedAt    → Última actualización

Restricción: @@unique([fromCurrency, toCurrency])
             No puede haber dos tasas para el mismo par de monedas.

Relaciones: Ninguna (tabla de referencia consultada en código)

Ejemplo de datos:
  USD → BOB  rate: 6.96
  USD → PEN  rate: 3.70
  USD → ARS  rate: 900.00

Uso en el flujo de recargas:
  Package.amountUsd × ExchangeRate.rate = monto en moneda local
  Ej: 10 USD × 6.96 = 69.60 BOB acreditados al Wallet

────────────────────────────────────────
USER
────────────────────────────────────────
Propósito: Modelo base SOLO para autenticación. Cualquier persona que
use la app tiene un User. No tiene rol fijo — cualquier usuario puede
postularse a ser provider cuando quiera (filosofía Facebook).
Los datos personales viven en Profile.

Campos:
  - email               → Correo para iniciar sesión (único, opcional si se
                          registra con teléfono, vacío "" para cuentas Google)
  - phone               → Teléfono para iniciar sesión (único, opcional si se
                          registra con email). Diferente a Contact.phone —
                          este es credencial de login, no número de contacto.
                          El frontend lo arma: Country.dialCode + número
  - username            → Elegido por el usuario al registrarse.
                          Solo letras y números, sin espacios.
                          Si ya existe, el backend sugiere 3 alternativas.
  - password            → Hash bcrypt de la contraseña
                          (vacío "" para cuentas Google — solo entran con OAuth)
  - isActive            → false = cuenta bloqueada por el admin
  - fcmToken            → Token Firebase para notificaciones push en tiempo real
  - resetPasswordToken  → Hash del código de recuperación de contraseña (6 dígitos)
  - resetPasswordExpiry → Expiración del código de recuperación (15 minutos)
  - createdAt           → Fecha de creación
  - updatedAt           → Última actualización

Login soporta 4 opciones:
  1. email + contraseña
  2. username + contraseña
  3. teléfono + contraseña  (User.phone)
  4. Google OAuth (flujo de 2 pasos para usuarios nuevos)

Regla: el usuario debe tener al menos email O phone al registrarse.

Relaciones:
  - profile          → Datos personales y wallet (1 a 1)
  - providerServices → Especialidades en las que actúa como provider (1 a N)
  - sentMessages     → Mensajes enviados en servicios (1 a N)
  - clientServices   → Servicios solicitados (1 a N)
  - ratings          → Calificaciones dadas a providers (1 a N)
  - historyViews     → Historias vistas (1 a N)
  - savedProviders   → Especialidades guardadas como favoritas (1 a N)
  - savedPosts       → Publicaciones guardadas como favoritas (1 a N)
  - postLikes        → Likes dados a publicaciones (1 a N)
  - postComments     → Comentarios escritos en publicaciones (1 a N)
  - reelLikes        → Likes dados a reels (1 a N)
  - reelComments     → Comentarios escritos en reels (1 a N)


────────────────────────────────────────
PROFILE
────────────────────────────────────────
Propósito: Perfil único de cada usuario. Se crea automáticamente al
registrarse. Contiene datos personales, foto de perfil y la billetera
para pagar servicios. Cualquier usuario puede también ser provider —
el perfil es el mismo para ambos roles (no hay ProviderProfile separado).

Campos:
  - userId         → FK a User (1 a 1, único)
  - countryIso     → FK a Country — define la moneda del Wallet.
                     Se elige al registrarse y NUNCA cambia.
  - firstName      → Nombre del usuario (nullable — se completa en el paso 2)
  - lastName       → Apellido del usuario (nullable — se completa en el paso 2)
  - avatar         → URL pública de la foto de perfil en Cloudinary
  - avatarPublicId → public_id de Cloudinary para reemplazar o eliminar la foto
  - referralCode   → Código único para referir a otros usuarios
  - lastLoginAt    → Último inicio de sesión — se actualiza en cada login
  - createdAt      → Fecha de creación
  - updatedAt      → Última actualización

Relaciones:
  - wallet   → Billetera para pagar servicios (1 a 1)
  - contacts → Números de contacto visibles del usuario (1 a N, máx 3)
  - user     → Pertenece a un User (N a 1)
  - country  → Pertenece a un Country — define la moneda del wallet (N a 1)


────────────────────────────────────────
CONTACT
────────────────────────────────────────
Propósito: Números de contacto visibles del usuario en su perfil.
Un usuario puede tener hasta 3 números. Separado de Profile para
permitir múltiples registros sin campos nullable.

Diferencia con User.phone:
  User.phone    → credencial de login, único global, máx 1
  Contact.phone → número visible en el perfil, hasta 3, informativo

Campos:
  - profileId → FK a Profile — número pertenece a un solo usuario
  - phone     → Número de teléfono único en toda la plataforma
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - profile → Pertenece a un Profile (N a 1)


────────────────────────────────────────
PROVIDERCONTACT
────────────────────────────────────────
Propósito: Números de contacto del provider por especialidad.
Una especialidad puede tener múltiples números. Separado de
ProviderService para permitir varios registros por especialidad.

Campos:
  - providerServiceId → FK a ProviderService
  - phone             → Número de teléfono único en toda la plataforma
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (N a 1)


────────────────────────────────────────
PROVIDERSERVICE
────────────────────────────────────────
Propósito: Especialidad de un usuario como provider. Cualquier usuario
puede postularse a una o más especialidades sin restricción de rol.
Cada especialidad es independiente: tiene su propio estado de
verificación, calificación, wallet de ganancias, y portafolio de
contenido (historias, posts, reels, anuncios).

Modelo similar a Uber — un solo login, múltiples modalidades de trabajo.
Modelo similar a Facebook — un usuario puede crear múltiples "páginas".

Restricción: @@unique([userId, serviceTypeId])
             Un usuario no puede postularse dos veces a la misma especialidad.

Campos:
  - userId        → FK a User — usuario que se postuló
  - serviceTypeId → FK a ServiceType — tipo de servicio (Gasfitería, etc.)
  - countryIso    → FK a Country — define la moneda del ProviderWallet.
                    Se elige al postularse y NUNCA cambia.
  - status        → PENDING → APPROVED (puede trabajar) | REJECTED
  - available     → true = disponible para recibir trabajos ahora mismo
  - rating        → Promedio de calificaciones (0.0 a 5.0)
  - totalRatings  → Total de calificaciones (usado para calcular el promedio)
  - latitude      → Ubicación actual — para encontrar el provider más cercano
  - longitude     → Ubicación actual
  - createdAt     → Fecha de creación
  - updatedAt     → Última actualización

Relaciones:
  - user             → Pertenece a un User (N a 1)
  - serviceType      → Pertenece a un ServiceType (N a 1)
  - country          → Pertenece a un Country — define moneda del wallet (N a 1)
  - providerWallet   → Billetera de ganancias (1 a 1)
  - profile          → Perfil de presentación visible para clientes (1 a 1)
  - documents        → Documentos y certificados de verificación (1 a N)
  - providerContacts → Números de contacto de esta especialidad (1 a N)
  - services         → Servicios atendidos con esta especialidad (1 a N)
  - histories        → Historias publicadas — expiran en 24h (1 a N)
  - posts            → Publicaciones permanentes tipo Facebook (1 a N)
  - reels            → Videos cortos tipo TikTok/Reels (1 a N)
  - anuncios         → Anuncios pagados para aparecer destacado (1 a N)
  - savedBy          → Usuarios que guardaron esta especialidad como favorita (1 a N)


────────────────────────────────────────
PROVIDERSERVICEPROFILE
────────────────────────────────────────
Propósito: Datos de PRESENTACIÓN de una especialidad. Contiene todo
lo que el cliente ve al buscar un provider: bio, experiencia, foto,
nombre personalizado. El provider puede editarlo libremente.
Se crea al postularse junto con ProviderService.

Separación de responsabilidades:
  ProviderService        → datos operativos (estado, ubicación, rating, moneda)
  ProviderServiceProfile → datos de presentación (bio, foto, nombre personalizado)

Campos:
  - providerServiceId    → FK a ProviderService (1 a 1, único)
  - bio                  → Descripción profesional para esta especialidad
  - experience           → Años de experiencia en esta especialidad
  - coverageRadius       → Radio máximo de cobertura en kilómetros
  - commercialName       → Nombre del negocio o marca personal (opcional)
  - customServiceName    → Nombre personalizado del servicio
                           (ej: "Plomería Express", "Electricista 24h")
  - servicePhoto         → URL pública de la foto del servicio en Cloudinary
  - servicePhotoPublicId → public_id de Cloudinary para reemplazar o eliminar
  - createdAt            → Fecha de creación
  - updatedAt            → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (1 a 1)

────────────────────────────────────────
WALLET
────────────────────────────────────────
Propósito: Billetera del usuario para pagar servicios.
Recibe recargas aprobadas por el admin.
La moneda se fija al registrarse según el país elegido y NUNCA cambia.

Campos:
  - profileId → FK a Profile (1 a 1, único)
  - currency  → Moneda fija del wallet (ej: BOB, USD, PEN).
                Definida por Profile.countryIso al registrarse.
  - balance   → Saldo disponible para pagar servicios
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - profile   → Pertenece a un Profile (1 a 1)
  - recharges → Historial de recargas de saldo (1 a N)


────────────────────────────────────────
PROVIDERWALLET
────────────────────────────────────────
Propósito: Billetera de ganancias del provider por especialidad.
Acumula las ganancias de servicios completados y financia los anuncios.
Cada especialidad tiene su propia wallet independiente.
La moneda se fija al postularse según el país elegido y NUNCA cambia.

Campos:
  - providerServiceId → FK a ProviderService (1 a 1, único)
  - currency          → Moneda fija del wallet (ej: BOB, USD, PEN).
                        Definida por ProviderService.countryIso al postularse.
  - balance           → Ganancias acumuladas disponibles para retirar
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (1 a 1)
  - withdrawals     → Historial de retiros de ganancias (1 a N)
  - purchases       → Historial de recargas compradas (ProviderPurchase) (1 a N)
  - anuncios        → Anuncios financiados desde esta billetera (1 a N)


────────────────────────────────────────
PACKAGE
────────────────────────────────────────
Propósito: Paquetes de recarga creados por el admin. Los usuarios los
compran para recargar su Wallet; los providers los compran para recargar
su ProviderWallet. El precio siempre está en USD y se convierte a la
moneda local usando ExchangeRate al acreditar el saldo.

Campos:
  - name      → Nombre del paquete (ej: "Paquete Básico", "Paquete Pro")
  - amountUsd → Precio base en USD — se convierte a moneda local al acreditar
  - active    → false = no aparece en la app para los usuarios
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - recharges         → Recargas de usuarios que usaron este paquete (1 a N)
  - providerPurchases → Compras de providers que usaron este paquete (1 a N)


────────────────────────────────────────
RECHARGE
────────────────────────────────────────
Propósito: Registro de cada recarga de saldo que hace un usuario.
Flujo: usuario elige Package → Recharge (PENDING) →
admin aprueba → Wallet.balance += package.amountUsd × ExchangeRate.rate

Campos:
  - walletId  → FK a Wallet — wallet del usuario que recarga
  - packageId → FK a Package — paquete elegido por el usuario
  - status    → PENDING | APPROVED | REJECTED
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - wallet  → Pertenece a un Wallet (N a 1)
  - package → Pertenece a un Package (N a 1)


────────────────────────────────────────
PROVIDERPURCHASE
────────────────────────────────────────
Propósito: Registro de cada recarga de saldo que hace un provider.
El provider compra un paquete para acreditar saldo en el ProviderWallet
de una especialidad específica (para financiar anuncios o uso futuro).
Flujo: provider elige Package → ProviderPurchase (PENDING) →
admin aprueba → ProviderWallet.balance += package.amountUsd × ExchangeRate.rate

Campos:
  - providerWalletId → FK a ProviderWallet — wallet de la especialidad
  - packageId        → FK a Package — paquete elegido
  - status           → PENDING | APPROVED | REJECTED (usa RechargeStatus)
  - createdAt        → Fecha de creación
  - updatedAt        → Última actualización

Nota: Usa el mismo enum RechargeStatus que Recharge porque los estados
      son idénticos. Son tablas separadas porque apuntan a wallets distintos:
        Recharge         → walletId        → Wallet
        ProviderPurchase → providerWalletId → ProviderWallet

Relaciones:
  - providerWallet → Pertenece a un ProviderWallet (N a 1)
  - package        → Pertenece a un Package (N a 1)


────────────────────────────────────────
WITHDRAWAL
────────────────────────────────────────
Propósito: Solicitud de retiro de ganancias del provider hacia su cuenta
bancaria. El retiro se hace por especialidad (cada una tiene su wallet).
Flujo: provider solicita → Withdrawal (PENDING) →
admin aprueba → ProviderWallet.balance -= amount

Campos:
  - providerWalletId → FK a ProviderWallet — wallet de donde se retira
  - amount           → Monto a retirar en la moneda del wallet
  - bankName         → Nombre del banco destino
  - accountNumber    → Número de cuenta bancaria destino
  - status           → PENDING | APPROVED | REJECTED
  - createdAt        → Fecha de creación
  - updatedAt        → Última actualización

Relaciones:
  - providerWallet → Pertenece a un ProviderWallet (N a 1)


────────────────────────────────────────
HISTORY
────────────────────────────────────────
Propósito: Historia que el provider sube para mostrar su trabajo,
estilo WhatsApp/Instagram Stories. Expira automáticamente a las 24h —
un cron job la elimina y borra el media de Cloudinary usando mediaPublicId.

Campos:
  - providerServiceId → FK a ProviderService — especialidad que publicó
  - mediaUrl          → URL pública del media en Cloudinary
  - mediaPublicId     → public_id de Cloudinary para eliminar al expirar
  - mediaType         → IMAGE | VIDEO
  - caption           → Texto opcional superpuesto sobre la historia
  - expiresAt         → Fecha de expiración (createdAt + 24h)
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (N a 1)
  - historyViews    → Registro de usuarios que vieron esta historia (1 a N)


────────────────────────────────────────
HISTORYVIEW
────────────────────────────────────────
Propósito: Registra cada vez que un usuario ve una historia.
Sirve para mostrar el conteo de vistas y evitar duplicados.

Campos:
  - historyId → FK a History — historia que fue vista
  - userId    → FK a User — usuario que la vio
  - viewedAt  → Momento exacto en que se vio

Restricción: @@unique([historyId, userId])
             Un usuario no puede contar como vista dos veces.

Relaciones:
  - history → Pertenece a una History (N a 1)
  - user    → Pertenece a un User (N a 1)


────────────────────────────────────────
POST
────────────────────────────────────────
Propósito: Publicación permanente del provider, estilo Facebook.
Puede tener texto y múltiples fotos o videos en carrusel.
Los usuarios pueden dar like, comentar y guardarla como favorita.

Campos:
  - providerServiceId → FK a ProviderService — especialidad que publicó
  - content           → Texto de la publicación (opcional si hay media)
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (N a 1)
  - media           → Fotos y videos en carrusel (1 a N)
  - likes           → Likes de usuarios (1 a N)
  - comments        → Comentarios de usuarios (1 a N)
  - savedPosts      → Usuarios que guardaron esta publicación (1 a N)


────────────────────────────────────────
POSTMEDIA
────────────────────────────────────────
Propósito: Media (foto o video) adjunta a una publicación en carrusel.
Un Post puede tener múltiples medias ordenadas por el campo "order".

Campos:
  - postId        → FK a Post
  - mediaUrl      → URL pública del media en Cloudinary
  - mediaPublicId → public_id de Cloudinary para eliminar el media
  - mediaType     → IMAGE | VIDEO
  - order         → Orden del media en el carrusel (1, 2, 3...)
  - createdAt     → Fecha de creación
  - updatedAt     → Última actualización

Relaciones:
  - post → Pertenece a un Post (N a 1)


────────────────────────────────────────
POSTLIKE
────────────────────────────────────────

Un usuario solo puede dar un like por publicación.

Campos:
  - postId    → FK a Post — publicación que recibió el like
  - userId    → FK a User — usuario que dio el like
  - createdAt → Fecha del like

Restricción: @@unique([postId, userId])

Relaciones:
  - post → Pertenece a un Post (N a 1)
  - user → Pertenece a un User (N a 1)

SAVEDPOST

────────────────────────────────────────
SAVEDPROVIDER
────────────────────────────────────────
Propósito: Especialidad de provider guardada como favorita,
estilo "contacto favorito" de Uber. Apunta a User (no Profile)
para que cualquier rol pueda guardar favoritos a futuro.

Campos:
  - userId            → FK a User — usuario que guardó
  - providerServiceId → FK a ProviderService — especialidad guardada
  - createdAt         → Fecha en que se guardó
  - updatedAt         → Última actualización

Restricción: @@unique([userId, providerServiceId])

Relaciones:
  - user            → Pertenece a un User (N a 1)
  - providerService → Pertenece a un ProviderService (N a 1)


────────────────────────────────────────
REEL
────────────────────────────────────────
Propósito: Video corto vertical del provider, estilo TikTok / Instagram
Reels. Independiente de Post — tiene su propia sección en la app con
thumbnail y contador de vistas.

Diferencia con Post:
  Post → publicación permanente, texto + carrusel de fotos/videos
  Reel → video corto vertical, thumbnail, sección propia, contador de vistas

Campos:
  - providerServiceId → FK a ProviderService — especialidad que publicó
  - videoUrl          → URL pública del video en Cloudinary
  - videoPublicId     → public_id de Cloudinary para eliminar el video
  - thumbnailUrl      → URL pública del thumbnail (primer frame) en Cloudinary
  - thumbnailPublicId → public_id de Cloudinary para eliminar el thumbnail
  - caption           → Texto opcional sobre el reel
  - duration          → Duración del video en segundos (máx recomendado: 60)
  - views             → Contador de reproducciones acumuladas
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - providerService → Pertenece a un ProviderService (N a 1)
  - likes           → Likes de usuarios (1 a N)
  - comments        → Comentarios de usuarios (1 a N)


────────────────────────────────────────
REELLIKE
────────────────────────────────────────
Propósito: Like de un usuario a un reel.
Un usuario solo puede dar un like por reel.

Campos:
  - reelId    → FK a Reel — reel que recibió el like
  - userId    → FK a User — usuario que dio el like
  - createdAt → Fecha del like

Restricción: @@unique([reelId, userId])

Relaciones:
  - reel → Pertenece a un Reel (N a 1)
  - user → Pertenece a un User (N a 1)

REELCOMMENT
────────────────────────────────────────
Propósito: Comentario de un usuario en un reel.

Campos:
  - reelId    → FK a Reel — reel comentado
  - userId    → FK a User — usuario que comentó
  - content   → Texto del comentario
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - reel → Pertenece a un Reel (N a 1)
  - user → Pertenece a un User (N a 1)


────────────────────────────────────────
ANUNCIO
────────────────────────────────────────
Propósito: Anuncio pagado por el provider para destacar su especialidad
en los resultados de búsqueda. Modelo prepago — el saldo se descuenta
del ProviderWallet al crear el anuncio.
El cron job lo desactiva automáticamente al llegar a endDate.

Flujo:
  Provider crea anuncio → ProviderWallet.balance -= budget
  → Anuncio activo desde startDate hasta endDate
  → Cron job: active = false al llegar endDate
  → Providers con anuncio activo aparecen primero en búsquedas

Campos:
  - providerWalletId  → FK a ProviderWallet — rastro financiero
                        (de qué wallet salió el dinero)
  - providerServiceId → FK a ProviderService — rastro comercial
                        (qué especialidad se está promocionando)
  - title             → Título del anuncio
  - description       → Texto descriptivo (opcional)
  - mediaUrl          → URL Cloudinary del media del anuncio (opcional)
  - mediaPublicId     → public_id de Cloudinary para eliminar el media
  - mediaType         → IMAGE | VIDEO (null si no hay media)
  - budget            → Saldo descontado del ProviderWallet al crear
  - startDate         → Fecha y hora de inicio del anuncio
  - endDate           → Fecha y hora de fin — cron job desactiva al llegar
  - active            → false = expirado o desactivado manualmente
  - impressions       → Veces que se mostró en resultados de búsqueda
  - clicks            → Veces que un usuario tocó el anuncio
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Nota: providerWalletId y providerServiceId apuntan al mismo usuario
      pero cumplen roles distintos y son necesarios por auditoría.

Relaciones:
  - providerWallet  → Pertenece a un ProviderWallet — origen del pago (N a 1)
  - providerService → Pertenece a un ProviderService — lo que se promociona (N a 1)


────────────────────────────────────────
SERVICE
────────────────────────────────────────
Propósito: Núcleo del negocio. Representa un trabajo solicitado por
un usuario a un provider. Conecta al usuario con una especialidad.
Ciclo de vida: SEARCHING → ASSIGNED → ON_WAY → IN_PROGRESS → COMPLETED | CANCELLED

Campos:
  - clientId          → FK a User — usuario que solicitó el servicio
  - providerServiceId → FK a ProviderService — especialidad asignada
                        (null mientras está en estado SEARCHING)
  - serviceTypeId     → FK a ServiceType — tipo de servicio solicitado
  - description       → Descripción del problema o trabajo a realizar
  - address           → Dirección física donde se realiza el servicio
  - latitude          → Latitud para mostrar en el mapa
  - longitude         → Longitud para mostrar en el mapa
  - status            → Estado actual del servicio
  - price             → Precio acordado entre usuario y provider
                        (null en estado SEARCHING — aún no se acordó)
  - commission        → Comisión cobrada por la plataforma
                        (price × SystemConfig.commissionRate, null en SEARCHING)
  - createdAt         → Fecha de creación
  - updatedAt         → Última actualización

Relaciones:
  - client          → Pertenece a un User — quien solicitó (N a 1)
  - providerService → Pertenece a un ProviderService — quien atiende (N a 1, nullable)
  - serviceType     → Pertenece a un ServiceType (N a 1)
  - messages        → Chat entre usuario y provider (1 a N)
  - calls           → Llamadas de voz o video durante el servicio (1 a N)
  - rating          → Calificación del usuario al finalizar (1 a 1, único)


────────────────────────────────────────
MESSAGE
────────────────────────────────────────
Propósito: Mensaje del chat dentro de un servicio activo.
Usuario y provider se comunican a través de texto, imágenes o audios.

Campos:
  - serviceId → FK a Service — servicio al que pertenece
  - senderId  → FK a User — usuario que envió el mensaje
  - content   → Texto del mensaje o URL del archivo en Cloudinary
  - type      → TEXT | IMAGE | AUDIO
  - read      → true = el receptor ya leyó el mensaje
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - service → Pertenece a un Service (N a 1)
  - sender  → Pertenece a un User (N a 1)


────────────────────────────────────────
CALL
────────────────────────────────────────
Propósito: Registro de llamadas de voz o video durante un servicio.

Campos:
  - serviceId  → FK a Service — servicio en el que ocurrió la llamada
  - callerId   → FK a User — usuario que inició la llamada
  - receiverId → FK a User — usuario que recibió la llamada
  - type       → VOICE | VIDEO
  - status     → INITIATED | ACCEPTED | REJECTED | ENDED | MISSED
  - startedAt  → Momento en que el receptor aceptó
  - endedAt    → Momento en que terminó
  - duration   → Duración en segundos (calculado al terminar)
  - createdAt  → Fecha de creación
  - updatedAt  → Última actualización

Relaciones:
  - service → Pertenece a un Service (N a 1)


────────────────────────────────────────
RATING
────────────────────────────────────────
Propósito: Calificación del usuario al provider después de completar
un servicio. Solo una calificación por servicio.
Al crearse recalcula automáticamente ProviderService.rating.

Fórmula: rating = (rating × totalRatings + score) / (totalRatings + 1)

Campos:
  - serviceId → FK a Service (único — un solo Rating por servicio)
  - clientId  → FK a User — usuario que calificó
  - score     → Puntuación del 1 al 5
  - comment   → Comentario opcional
  - createdAt → Fecha de creación
  - updatedAt → Última actualización

Relaciones:
  - service → Pertenece a un Service (1 a 1)
  - client  → Pertenece a un User (N a 1)

4. FLUJOS PRINCIPALES DEL NEGOCIO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

REGISTRO (2 pasos, cualquier persona):
  Paso 1 → POST /auth/register
    User (email O phone) + username + countryIso
    → Profile (countryIso, firstName: null, lastName: null)
    → Wallet (currency = Country.currency, balance: 0)
  Paso 2 → PUT /profile/complete
    → Profile.firstName, lastName
    → Contact[] (teléfonos de contacto opcionales, máx 3)

REGISTRO CON GOOGLE (un solo flujo):
  POST /auth/google → si email existe: login directo
                   → si es nuevo: devuelve tempToken + datos de Google
  POST /auth/google/complete → username + countryIso
    → crea User + Profile + Wallet
    → firstName/lastName vienen de Google, no requiere Paso 2

LOGIN (4 opciones):
  1. email + contraseña    → POST /auth/login
  2. username + contraseña → POST /auth/login
  3. teléfono + contraseña → POST /auth/login  (User.phone)
  4. Google OAuth          → POST /auth/google
  → Actualiza Profile.lastLoginAt

POSTULACIÓN A ESPECIALIDAD (cualquier usuario):
  Usuario elige ServiceType + countryIso
  → ProviderService (PENDING)
  → ProviderWallet (currency = Country.currency, balance: 0)
  → ProviderServiceProfile (bio, experience, etc.) — opcional
  → ProviderDocument[] (certificados) — opcional
  → ProviderContact[] (teléfonos de la especialidad) — opcional
  Admin aprueba → ProviderService.status = APPROVED

RECARGA DE SALDO (USUARIO):
  GET /packages → usuario elige paquete
  POST /wallet/recharge → Recharge (PENDING)
  Admin aprueba:
    tasa = ExchangeRate WHERE fromCurrency='USD' AND toCurrency=wallet.currency
    Wallet.balance += package.amountUsd × tasa
  Ej: 10 USD × 6.96 = 69.60 BOB

RECARGA DE SALDO (PROVIDER — para financiar anuncios):
  GET /packages → provider elige paquete + especialidad
  POST /wallet/purchase → ProviderPurchase (PENDING)
  Admin aprueba:
    tasa = ExchangeRate WHERE fromCurrency='USD' AND toCurrency=wallet.currency
    ProviderWallet.balance += package.amountUsd × tasa

SOLICITUD DE SERVICIO:
  Usuario crea Service (SEARCHING)
  Sistema busca ProviderService donde:
    status = APPROVED AND available = true
    serviceTypeId = el solicitado
    trabajos activos < SystemConfig.maxConcurrentJobs
    distancia < coverageRadius (por lat/lng)
  Provider acepta → ASSIGNED → ON_WAY → IN_PROGRESS → COMPLETED

PAGO AL COMPLETAR:
  Wallet.balance         -= price
  ProviderWallet.balance += (price - commission)
  commission = price × SystemConfig.commissionRate

CALIFICACIÓN:
  Service COMPLETED → Usuario crea Rating (score 1-5)
  ProviderService.rating = (rating × totalRatings + score) / (totalRatings + 1)
  ProviderService.totalRatings += 1

RETIRO DE GANANCIAS:
  Provider solicita Withdrawal (amount, bankName, accountNumber)
  Admin aprueba → ProviderWallet.balance -= amount

CONTENIDO DEL PROVIDER:
  Historia → expira en 24h → cron job la elimina + borra media de Cloudinary
  Post     → permanente, texto + carrusel, likes, comentarios, guardado
  Reel     → video corto vertical, thumbnail, vistas, likes, comentarios

ANUNCIO PAGADO:
  Provider crea Anuncio → ProviderWallet.balance -= budget
  Activo hasta endDate → aparece destacado en búsquedas
  Cron job: active = false al llegar endDate
  Métricas: impressions + clicks

RECUPERAR CONTRASEÑA:
  Usuario ingresa email → código de 6 dígitos al correo (válido 15 min)
  Máximo 5 intentos fallidos → código invalidado
  Usuario ingresa código → nueva contraseña

5. DECISIONES DE DISEÑO IMPORTANTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. UN SOLO PERFIL SIN ROLES FIJOS (FILOSOFÍA FACEBOOK)
   No hay ClientProfile + ProviderProfile separados.
   Existe un solo Profile por usuario. Cualquier usuario puede
   postularse a ser provider sin cambiar de cuenta ni de rol.
   El "rol de provider" lo define tener un ProviderService aprobado.

2. DOS TIPOS DE TELÉFONO CON PROPÓSITOS DISTINTOS
   User.phone    → credencial de login, único global, máx 1
   Contact.phone → número de contacto visible en el perfil, hasta 3
   El frontend usa Country.dialCode para armar User.phone.

3. MONEDA FIJA EN WALLETS
   Wallet.currency y ProviderWallet.currency se asignan al crear
   el wallet y NUNCA cambian. Esto evita inconsistencias si el admin
   modifica Country.currency en el futuro.

4. PAQUETES EN USD CON CONVERSIÓN AUTOMÁTICA
   Package.amountUsd siempre en USD.
   ExchangeRate convierte al momento de acreditar.
   Un solo catálogo de paquetes sirve para todos los países.

5. REGISTRO EN DOS PASOS
   Paso 1 crea credenciales + país + wallet (mínimo para operar).
   Paso 2 completa nombre, apellido y teléfonos de contacto.
   Permite explorar la app antes de completar el perfil.

6. CONTENIDO LIGADO A ESPECIALIDAD, NO AL PERFIL
   History, Post, Reel y Anuncio apuntan a ProviderService.
   Cada especialidad tiene su propio portafolio independiente.
   Ej: Juan el Electricista y Juan el Gasfitero tienen
   historias, posts y reels completamente separados.

7. PRICE Y COMMISSION SON NULLABLE EN SERVICE
   Ambos son Float? porque en SEARCHING el precio aún
   no se ha acordado entre usuario y provider.

8. ANUNCIO TIENE DOS FKs AL MISMO PROVIDER
   providerWalletId  → rastro financiero (de dónde salió el dinero)
   providerServiceId → rastro comercial (qué especialidad se promociona)
   Ambas son necesarias por semántica y auditoría.

9. SAVEDPROVIDER Y SAVEDPOST USAN USER (NO PROFILE)
   Apuntan directamente a User para que cualquier usuario pueda
   guardar favoritos independientemente de su estado como provider.
