1. ¿QUÉ ES HOOMFIX?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Hoomfix es un marketplace on-demand de servicios del hogar.
Conecta a personas que necesitan un técnico (gasfitero, electricista,
etc.) con providers verificados disponibles en su zona.

Tipo de software : Marketplace de servicios (platform business)
Modelo de negocio: Comisión por transacción (take-rate model)
Plataforma       : Aplicación móvil + Backend API REST
Inspiración      : Uber, TaskRabbit, Facebook


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2. MODELO DE USUARIOS — FILOSOFÍA FACEBOOK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Hoomfix no tiene roles fijos al registrarse. Cualquier usuario
puede solicitar servicios Y ofrecer servicios, igual que en Facebook
cualquier persona puede usar la red social Y crear una página de negocio.

  Registro → cuenta única (User + Profile + Wallet)
           → ve el feed de providers
           → puede solicitar servicios

  Quiero ofrecer servicios → postula a una especialidad
                           → admin aprueba
                           → ya puede recibir trabajos

Un mismo usuario puede:
  ✓ Solicitar un gasfitero para su casa
  ✓ Ofrecer servicios de electricidad con su propia especialidad
  ✓ Tener múltiples especialidades (Gasfitería + Electricidad)
  ✓ Publicar historias, posts y reels de su trabajo
  ✓ Guardar providers favoritos
  ✓ Dar likes y comentar publicaciones

3. ¿CÓMO GANA DINERO HOOMFIX?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Por cada servicio completado, Hoomfix cobra una comisión:

  Precio del servicio: Bs. 100
  Comisión (15%):      Bs. 15   → va a Hoomfix
  Ganancia provider:   Bs. 85   → va al ProviderWallet

El porcentaje de comisión es configurable desde SystemConfig.

SISTEMA DE MONEDAS:
  Los paquetes de recarga se almacenan en USD (moneda base).
  Al mostrarlos al usuario se convierten automáticamente a su
  moneda local usando la tabla ExchangeRate.

  Ejemplo:
    Paquete Pro = 10 USD
    Usuario Bolivia → 10 × 6.96 = 69.60 BOB
    Usuario Perú    → 10 × 3.70 = 37.00 PEN

  El admin gestiona los tipos de cambio desde el panel (ExchangeRate).
  Un solo catálogo de paquetes sirve para todos los países.

5.1 REGISTRO Y AUTENTICACIÓN
────────────────────────────────────────

REGISTRO (2 pasos, cualquier persona):

  Paso 1 — Crear cuenta:
    → El usuario elige registrarse con email O con teléfono
    → Elige su username (único, solo letras y números)
    → Elige su país del combo box (define la moneda del Wallet)
    → Se crea User + Profile + Wallet (saldo: 0, moneda por país)

    Con email:    { email, username, countryIso, password }
    Con teléfono: { phone, username, countryIso, password }
      El frontend arma el phone: Country.dialCode + número
      Ej: "+591" + "70012345" = "+59170012345"

  Paso 2 — Completar perfil:
    → Completa firstName, lastName
    → Agrega teléfonos de contacto opcionales (máx 3)
      Nota: estos son teléfonos visibles en el perfil,
            distintos al User.phone que es para login

LOGIN — 4 formas:
  1. email + contraseña
  2. username + contraseña
  3. teléfono + contraseña   ← User.phone
  4. Google OAuth

REGISTRO CON GOOGLE (un solo flujo):
  → El app abre el selector de cuentas de Google
  → Si el email ya existe → login directo
  → Si es nuevo → elige username + país → cuenta creada
  → firstName y lastName vienen de Google (no requiere Paso 2)

RECUPERAR CONTRASEÑA:
  → Código de 6 dígitos enviado al email
  → Válido 15 minutos, máximo 5 intentos


────────────────────────────────────────
5.2 POSTULACIÓN A ESPECIALIDAD
────────────────────────────────────────

Cualquier usuario puede postularse a ofrecer un servicio en cualquier
momento, sin necesidad de un rol especial.
Modelo similar a Uber — un solo login, múltiples modalidades de trabajo.

Cada especialidad es completamente independiente — tiene su propio
perfil de presentación, wallet de ganancias, estado de verificación,
disponibilidad y contenido (posts, reels, historias, anuncios).

Pasos:
  1. Usuario elige un ServiceType disponible
     (Gasfitería, Electricidad, Carpintería, etc.)

  2. Elige el país donde va a trabajar esa especialidad
     El país define la moneda del ProviderWallet (fija para siempre)

  3. Completa el perfil de presentación (opcional al postularse):
       - Bio profesional
       - Años de experiencia
       - Radio máximo de cobertura en kilómetros
       - Nombre comercial (opcional)
       - Nombre personalizado del servicio (opcional)
         Ej: "Plomería Express", "Electricista 24h"
       - Foto del servicio (opcional, subida vía Cloudinary)

  4. Agrega teléfonos de contacto para esta especialidad (opcional)

  5. Sube documentos de verificación (opcional al postularse):
       - Certificados, títulos, carnets profesionales
       - Cada documento se guarda en ProviderDocument

  6. La especialidad queda en estado PENDING
     → Admin recibe notificación para revisar

  7. Admin revisa los documentos y decide:
       - APPROVED → usuario puede activar disponibilidad
       - REJECTED → usuario debe corregir y volver a postularse

  8. Con estado APPROVED el usuario puede:
       - Activar/desactivar disponibilidad (available: true/false)
       - Actualizar ubicación en tiempo real (lat/lng)
       - Recibir solicitudes de clientes cercanos
       - Acumular ganancias en su ProviderWallet
       - Editar su perfil de presentación libremente
       - Publicar historias, posts, reels y anuncios

  9. Se crea ProviderWallet automáticamente al postularse (saldo: 0)

Restricción: Un usuario no puede postularse dos veces a la misma
especialidad (@@unique([userId, serviceTypeId])).


────────────────────────────────────────
5.3 CONTENIDO DEL PROVIDER
────────────────────────────────────────

Cada especialidad tiene su propio portafolio de contenido,
independiente de las otras especialidades del mismo usuario.

HISTORIA (estilo WhatsApp/Instagram Stories):
  → El provider sube una foto o video corto
  → Visible para todos los usuarios por 24 horas
  → El cron job la elimina automáticamente al expirar
  → Los usuarios pueden ver quién la vio (HistoryView)

POST (estilo Facebook):
  → Publicación permanente con texto y/o carrusel de fotos/videos
  → Los usuarios pueden dar like y comentar
  → Cualquier usuario puede guardarla como favorita (SavedPost)

REEL (estilo TikTok/Instagram Reels):
  → Video corto vertical con thumbnail
  → Duración máx recomendada: 60 segundos
  → Contador de reproducciones, likes y comentarios
  → Sección propia en la app

ANUNCIO PAGADO:
  → El provider paga con su ProviderWallet para aparecer
    destacado en los resultados de búsqueda
  → Modelo prepago: el saldo se descuenta al crear el anuncio
  → Activo desde startDate hasta endDate
  → El cron job lo desactiva automáticamente al expirar
  → Métricas: impresiones (veces mostrado) y clicks (veces tocado)


────────────────────────────────────────
5.4 RECARGA DE SALDO (WALLET DEL USUARIO)
────────────────────────────────────────

El usuario necesita saldo en su Wallet para pagar servicios.

Pasos:
  1. Usuario ve lista de paquetes disponibles (creados por admin)
     Los montos se muestran convertidos a la moneda del usuario
     Ej: 10 USD × 6.96 = 69.60 BOB para Bolivia
  2. Usuario elige un paquete → se crea Recharge (PENDING)
  3. Admin aprueba la recarga
  4. Backend calcula: amountUsd × ExchangeRate.rate = monto local
  5. Wallet.balance += monto local
  6. Usuario recibe notificación (push + email)


────────────────────────────────────────
5.5 RECARGA DE SALDO (PROVIDERWALLET)
────────────────────────────────────────

El provider puede comprar paquetes para recargar el saldo
de una especialidad específica (para financiar anuncios).

Pasos:
  1. Provider elige un paquete y la especialidad a recargar
     → se crea ProviderPurchase (PENDING)
  2. Admin aprueba la compra
  3. Backend calcula: amountUsd × ExchangeRate.rate = monto local
  4. ProviderWallet.balance += monto local
  5. Provider recibe notificación (push + email)


────────────────────────────────────────
5.6 SOLICITUD Y CICLO DE SERVICIO
────────────────────────────────────────

BÚSQUEDA DE PROVIDER:
  Sistema busca el ProviderService más cercano que cumpla:
  ✓ status = APPROVED
  ✓ available = true
  ✓ serviceTypeId = el solicitado por el usuario
  ✓ trabajos activos < SystemConfig.maxConcurrentJobs
  ✓ distancia al usuario < coverageRadius

CICLO DE VIDA:

  [SEARCHING]
      Usuario solicita servicio con descripción y ubicación
      Sistema notifica al provider cercano disponible
          ↓
  [ASSIGNED]
      Provider acepta el trabajo
      Usuario recibe notificación del provider asignado
          ↓
  [ON_WAY]
      Provider confirma que va en camino
      Usuario puede ver ubicación del provider en el mapa
          ↓
  [IN_PROGRESS]
      Provider llegó y comenzó el trabajo
      Usuario y provider pueden chatear y llamarse
          ↓
  [COMPLETED]
      Trabajo terminado
      → Pago procesado automáticamente
      → Usuario puede calificar el servicio

  En cualquier estado antes de COMPLETED:
      → [CANCELLED] por usuario o provider

COMUNICACIÓN DURANTE EL SERVICIO:
  - Chat en tiempo real (texto, imágenes, notas de voz)
  - Llamadas de voz o video


────────────────────────────────────────
5.7 PAGO AL COMPLETAR SERVICIO
────────────────────────────────────────

  precio_servicio = acordado entre usuario y provider
  comision        = precio_servicio × SystemConfig.commissionRate
  ganancia_neta   = precio_servicio - comision

  Wallet.balance         -= precio_servicio
  ProviderWallet.balance += ganancia_neta

  Si el usuario y provider usan monedas distintas:
  → Se consulta ExchangeRate para la conversión

  La moneda de cada wallet es fija (definida al crear el wallet):
  → Wallet del usuario  : según countryIso elegido al registrarse
  → ProviderWallet      : según countryIso elegido al postularse


────────────────────────────────────────
5.8 CALIFICACIÓN
────────────────────────────────────────

  Solo disponible cuando el servicio está COMPLETED.
  Un solo Rating por servicio.
  Puntuación del 1 al 5 + comentario opcional.

  Al calificar se recalcula automáticamente:
  ProviderService.rating =
    (rating_actual × totalRatings + nuevo_score) / (totalRatings + 1)


────────────────────────────────────────
5.9 RETIRO DE GANANCIAS (PROVIDER)
────────────────────────────────────────

  El provider solicita retirar el saldo de una especialidad específica.

  Pasos:
    1. Provider indica monto, banco y número de cuenta
    2. Se crea Withdrawal (PENDING)
    3. Admin aprueba el retiro
    4. ProviderWallet.balance -= amount
    5. Provider recibe notificación (push + email)

  Cada especialidad tiene su ProviderWallet independiente —
  el retiro se hace por especialidad, no de forma global.


────────────────────────────────────────
5.10 FAVORITOS
────────────────────────────────────────

  PROVIDERS FAVORITOS (SavedProvider):
    → El usuario guarda una especialidad como favorita
    → Puede guardar cualquier especialidad, sin importar su rol
    → Para encontrar rápido a su provider de confianza

  PUBLICACIONES GUARDADAS (SavedPost):
    → El usuario guarda publicaciones de providers
    → Estilo "Guardar" de Instagram
    → Para revisar después el trabajo de un provider

6. PANEL DE ADMINISTRACIÓN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El admin gestiona toda la plataforma:

  USUARIOS
    → Ver y bloquear/desbloquear cuentas (User.isActive)

  ESPECIALIDADES
    → Ver postulaciones pendientes
    → Revisar documentos subidos
    → Aprobar o rechazar especialidades (ProviderService)

  CATÁLOGO DE SERVICIOS
    → Crear nuevas categorías (ServiceType)
    → Activar o desactivar categorías

  PAÍSES
    → Ver lista de países (Country)
    → Activar o desactivar países del combo box
    → El dialCode se usa para armar teléfonos en el frontend

  TIPOS DE CAMBIO
    → Gestionar tasas de conversión (ExchangeRate)
    → Ej: USD → BOB = 6.96 / USD → PEN = 3.70
    → Se usan al acreditar recargas en la moneda local del usuario

  PAQUETES DE RECARGA
    → Crear paquetes con precio en USD (Package)
    → Activar o desactivar paquetes

  RECARGAS DE USUARIOS
    → Aprobar o rechazar solicitudes de recarga (Recharge)

  COMPRAS DE PROVIDERS
    → Aprobar o rechazar compras de paquetes (ProviderPurchase)

  RETIROS
    → Aprobar o rechazar solicitudes de retiro (Withdrawal)

  ANUNCIOS
    → Los anuncios se desactivan automáticamente vía cron job
    → El admin puede monitorear impresiones y clicks

  CONFIGURACIÓN GLOBAL (SystemConfig)
    → Ajustar comisión de la plataforma
    → Ajustar máximo de trabajos simultáneos por provider

8. STACK TECNOLÓGICO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Imágenes/docs : Cloudinary (con publicId para eliminar)
  Notificaciones: Firebase Cloud Messaging (FCM)
  Emails        : Nodemailer + Handlebars templates
  Documentación : Swagger (disponible en /docs)
  Países/monedas: Tabla Country gestionada por admin + seed inicial
  Tipos de cambio: Tabla ExchangeRate gestionada por admin
